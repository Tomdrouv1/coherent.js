<task type="auto">
  <name>Task 1: Create module tracker for HMR boundary detection</name>
  <files>packages/client/src/hmr/module-tracker.js</files>
  <action>
Create ModuleTracker class that tracks module graph and determines HMR boundaries:

1. **Module Registration:**
   - `modules` Map storing moduleId -> { accept, acceptDeps, dispose, prune, data }
   - `createHotContext(moduleId)` - returns hot context object for a module

2. **Hot Context API (Vite-compatible):**
   - `data` - persistent object that survives HMR updates (initially empty {})
   - `accept(callback)` - register self-update handler
   - `acceptDeps(deps, callback)` - register dependency update handler
   - `dispose(callback)` - register cleanup handler called before replacement
   - `prune(callback)` - register handler for when module is removed
   - `invalidate(message)` - signal that module cannot hot-update (propagate to importers)

3. **Boundary Detection:**
   - `canHotUpdate(moduleId)` - returns true if module has accept handler registered
   - `isHmrBoundary(moduleId)` - checks for accept handler OR __hmrBoundary export marker
   - Components with `data-coherent-component` are considered boundaries

4. **Update Execution:**
   - `executeDispose(moduleId)` - calls dispose callback if registered, passes data object
   - `executeAccept(moduleId, newModule)` - calls accept callback with new module
   - `executeAcceptDeps(moduleId, deps)` - calls acceptDeps callback with updated dependencies

5. **Export:**
   - Named export `ModuleTracker` class
   - Named export `moduleTracker` singleton instance

Per RESEARCH.md: Module boundary detection uses explicit __hmrBoundary export or data-coherent-component presence.
  </action>

<task type="auto">
  <name>Task 2: Create HMR client integrating all modules</name>
  <files>packages/client/src/hmr/client.js</files>
  <action>
Create HMRClient class that orchestrates WebSocket connection and HMR updates:

1. **Connection Management:**
   - `connect()` - establishes WebSocket to dev server (ws:// or wss:// based on page protocol)
   - Track `connected` boolean, `reconnectAttempts` counter, max 10 attempts
   - Exponential backoff with jitter: delay = min(1000 * 2^attempts + random(0-1000), 30000)
   - On open: log `[HMR] Connected`, reset attempts, update indicator('connected')
   - On close: update indicator('disconnected'), schedule reconnect
   - On reconnect after disconnect: full page reload (server may have restarted)

2. **Message Handling:**
   - Parse JSON messages, log `[HMR] message {type} {filePath}`
   - 'connected' - acknowledge connection
   - 'hmr-full-reload' / 'reload' - call location.reload()
   - 'hmr-component-update' / 'hmr-update' - call handleUpdate(data)
   - 'hmr-error' - call errorOverlay.show(data.error)

3. **Update Flow (handleUpdate):**
   - Capture state: stateCapturer.captureAll()
   - Get moduleId from message
   - If module has dispose handler: moduleTracker.executeDispose(moduleId), cleanupTracker.cleanup(moduleId)
   - Re-import module with cache-bust: `import(\`${path}?t=${Date.now()}\`)`
   - If module can hot-update: moduleTracker.executeAccept(moduleId, newModule)
   - Else: fall back to re-hydrate via autoHydrate (existing behavior)
   - Restore state: stateCapturer.restoreAll()
   - Hide error overlay (in case previous error now fixed)
   - Log `[HMR] Updated: {updateType} {filePath}`

4. **Error Handling:**
   - Wrap handleUpdate in try/catch
   - On error: errorOverlay.show({ message, file, line, column, frame, stack })
   - Parse error stack to extract file/line if possible

5. **Initialization:**
   - `initialize()` - call connect(), guard with window.__coherent_hmr_initialized
   - Auto-initialize when module loads (if window exists)

6. **Export:**
   - Named export `HMRClient` class
   - Named export `hmrClient` singleton instance

Follow RESEARCH.md Pattern 1 for connection handling. Integrate with all four modules from plans 01 and 02.
  </action>
